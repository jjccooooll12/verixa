# Verixa V2 Plan

This document is the focused v2 implementation plan for Verixa.

The goal is not to broaden the product. The goal is to make Verixa sharper as a warehouse-native data CI tool for source contracts, baseline-aware drift detection, and CI-friendly review workflows.

## Status Legend
- `[ ]` not started
- `[~]` in progress
- `[x]` completed
- `[!]` intentionally deferred

## Guardrails
- [x] Keep Verixa focused on source contracts and drift detection.
- [x] Keep connectors explicit and warehouse-specific.
- [x] Keep diff logic centralized.
- [x] Prefer deterministic outputs over opaque heuristics.
- [x] Treat policy engines as downstream consumers, not embedded rule engines.
- [x] Avoid turning Verixa into general CI, observability, or governance infrastructure.

## Milestone 1: CI and PR Workflow UX
- [x] Create a dedicated v2 plan and milestone list.
- [x] Add a stable normalized finding schema separate from console rendering.
- [x] Add stable finding fingerprints suitable for automation.
- [x] Add a `policy-v1` output format for diff-like commands.
- [x] Add a GitHub-friendly markdown summary format.
- [x] Add concise GitHub annotation output.
- [x] Group findings by source, severity, and change type in all reviewer-facing output.
- [x] Add explicit “new vs recurring vs resolved” lifecycle semantics.
- [x] Stabilize dotted finding codes for long-term automation compatibility.

## Milestone 2: Baseline Lifecycle Management
- [x] Add explicit baseline metadata manifests with baseline id, environment, reason, actor, git sha, and promotion metadata.
- [x] Add `verixa baseline status`.
- [x] Add `verixa baseline propose`.
- [x] Add `verixa baseline promote`.
- [x] Add `verixa baseline accept` for expected changes without blind overwrite.
- [x] Distinguish missing environment baseline vs missing per-source baseline explicitly.
- [x] Archive prior baselines on promotion.
- [x] Make stale baseline remediation more explicit and auditable.

## Milestone 3: Low-Noise Operation
- [x] Add a suppression file with expiry, owner, and reason.
- [x] Add per-finding-code severity overrides.
- [x] Add recurring finding tracking.
- [x] Add confidence metadata to normalized findings.
- [x] Add a clearer advisory mode for gradual rollout.
- [x] Surface expired suppressions in diagnostics.

## Milestone 4: History-Aware Drift Detection
- [x] Introduce optional snapshot-history storage for history-aware drift checks.
- [x] Add deterministic rolling-band drift checks for noisy metrics.
- [x] Support expected variability windows for row count, null rate, p50, and p95.
- [x] Add an explicit backfill mode for bursty sources.
- [x] Persist derived history bands in run artifacts for explainability.

## Milestone 5: Deeper dbt and Lineage Integration
- [x] Improve mapping from changed dbt files to affected sources.
- [x] Add downstream model impact summaries to findings.
- [x] Import owner and criticality metadata from dbt metadata when available.
- [x] Add PR-facing “what might break downstream” summaries.
- [x] Preserve deterministic fallback behavior when dbt metadata is incomplete.

## Milestone 6: Policy Composition
- [x] Introduce a stable policy export contract for findings.
- [x] Add normalized metadata for severity, source criticality, change type, baseline status, and lifecycle status.
- [x] Add example OPA integration docs and fixtures.
- [ ] Keep policy evaluation outside Verixa itself.

## Milestone 7: Cost-Aware Execution
- [x] Add execution modes: `cheap`, `bounded`, and `full`.
- [x] Add clearer estimated vs actual warehouse impact reporting.
- [x] Add cost-oriented source selection strategies for CI.
- [x] Add per-source scan confidence metadata when bounded modes are used.
- [x] Keep BigQuery and Snowflake cost behaviors explicit rather than forced into one abstraction.

## Milestone 8: Extensibility
- [x] Add custom check extension points.
- [x] Add finding enrichment hooks.
- [ ] Add custom renderer or output adapter hooks.
- [x] Add source metadata enrichment hooks.
- [x] Keep extension APIs narrow and typed.

## Notes
- Start with reviewer clarity and stable machine outputs before adding more checks.
- Baseline lifecycle is the next highest-leverage area after output normalization.
- History-aware drift detection should stay deterministic and opt-in.
- BigQuery now exposes budget-aware source-selection previews through `verixa cost --budget-bytes`.
- Diff-like and snapshot outputs now distinguish estimated BigQuery impact from actual Snowflake usage when Verixa can observe it safely.
- Changed-file targeting now records why a source was selected, with stable reason codes and confidence metadata for JSON, policy, and PR output.
- Extensions now support config-loaded `module:attribute` hooks for custom checks, finding enrichment, and source metadata enrichment.
- Renderer and output-adapter hooks remain intentionally deferred until the built-in PR and policy outputs stabilize further.
